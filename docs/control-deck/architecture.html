Chapter 01The Shape

Control Deck is not one thing. It is thirteen rooms in a house that was framed before any room was finished, and the work you see when you open a door is wildly uneven: some rooms are lived-in, some are drywalled, some are still drawn on paper tacked to a stud. The temptation — with a system this ambitious — is to describe the blueprint. The honest thing is to describe the house.

This chronicle does the honest thing. It walks through every surface the deck exposes — chat, terminal, four creative surfaces, five agent-infra tabs, the native shell, and the cross-cutting chrome — and names, for each: what actually works end-to-end today, what is wired but mute (types, routes, IPC handlers, UI slots with no caller), and what is still intent without scaffolding. No aspiration is scored as shipped. No shipped mechanism is underplayed.

The finding, in one sentence, before any detail: the deck is a mature frame with three genuinely deep rooms, four half-finished rooms, and six rooms that exist primarily as doors.

Chapter 02Chat — the one that carries the product

The chat surface is the most complete thing in the repository. It is also the one that most exposes the convention Control Deck has chosen: build the UI contract first, and outsource cognition to a subprocess. The subprocess in question is agent-go, a local Go binary on port 4243, and the deck's chat route (app/api/chat/route.ts) is an SSE proxy in front of it with a reducer on the other side.

User ─► ChatSurface ─► useAgentRun (reducer + SSE) useAgentRun ─POST /api/chat─► agent-go :4243 agent-go ─SSE─► /api/chat/route.ts ─► hub.publish(threadId, AGUIEvent) AGUIEvent ─► dispatchSSEEvent ─► agentRunReducer ─► AgentRunState AgentRunState.segments ─► ChatTimeline ─► TimelineSegments InterruptRequested ─► InterruptDialog ─POST─► agent-go Thread ─► SQLite (threads, messages) · Runs ─► SQLite (runs, events)

What ships. Three visual variants (safe, brave, radical) of the same reducer-driven surface. A full AG-UI state machine — SUBMIT → RUN_STARTED → TEXT_* / TOOL_* / REASONING_* → RUN_FINISHED/ERROR — dispatched at lib/hooks/useAgentRun.ts:369. Human-in-the-loop interrupts that round-trip through /api/chat/approve. Voice chat with push-to-talk, VAD, barge-in, and TTS auto-read wired end-to-end. Thread persistence with LLM-generated titles in SQLite.

What is wired but mute. The provider abstraction in lib/llm/providers.ts supports ten backends, has a runtime-override mechanism (setRuntimeProvider), and a system-prompt builder (buildSystemPrompt) that composes GPU stats + a GLYPH-compressed tool catalog + model-specific meta files. None of it is called in the chat path. The Go subprocess owns the system prompt; the deck's careful prompt machinery has no consumer. moby.txt, moby-lite.txt, and every file under lib/prompts/meta/ are unreferenced at runtime.

The experimental AG-UI dojo (lib/agui/experimental/, formerly lib/agui/dojo/) is explicit about being unwired: its index comment reads "Reference implementation — not wired into the main deck surface yet." Its hook (useAgui.ts) has exactly one non-test caller: a demo card.

The silent gap that matters most

When an InterruptRequested fires, the dialog renders and the user can approve. The approve HTTP call succeeds. But ChatSurface.onApprove (ChatSurface.tsx:1073-1080) only calls console.log and setPendingInterrupt(null) — no action dispatched to the reducer. There is no INTERRUPT_RESOLVED case. The InterruptResolved event that agent-go sends back falls through dispatchSSEEvent's default no-op branch. The run stays visually frozen in whatever phase it was in when the interrupt fired, even though the server has resumed.

Gaps

  1. High buildSystemPrompt authored, not called.

    The deck's carefully-composed system prompt (environment, GPU, tool catalog, model meta) has zero active callsites. agent-go owns the prompt. If the Go-side prompt diverges from the deck's intent, there is no reconciliation. The whole lib/prompts/ tree is dead from the main chat pipeline's perspective.

  2. High Interrupt approvals don't rehydrate run state.

    No INTERRUPT_RESOLVED action; no case for the InterruptResolved event; no phase transition after approve. Missing ~15 lines in the reducer and dispatcher.

  3. Medium Event types declared but silently dropped.

    CostIncurred is imported in the route but never emitted. StepStarted and StepFinished are emitted by agent-go but have no reducer case — they fall through dispatchSSEEvent's default. Inspector and timeline have nowhere to show step-level progress or accumulated cost, though the protocol carries it.

Chapter 03Terminal — solid spine, absent spirit

The terminal surface is a decent re-reading of the Amp/Zed/Goose sidebar-plus-dominant-pane pattern. A thin PTY sidecar (scripts/terminal-service.ts, Node + node-pty, raw WebSocket on port 4010) backs a launcher with three profiles — shell, claude, opencode — and an @wterm/react viewport.

TerminalPage └─► TerminalPane ├─► useTerminalSessions ─► terminalClient ─WS─► :4010 ─► node-pty ├─► Launcher (shell | claude | opencode) ├─► Session list (localStorage-persisted active id) ├─► SessionContextBox (NOW panel: OFFLINE | IDLE | EXITED | TASK STACK | NOW) └─► @wterm/react Terminal

What ships. Real PTYs. Singleton-per-label behaviour for claude and opencode (relaunch re-attaches). 1 MB scrollback replay on re-attach. Health polling every 5s. Alt+N / Alt+W / Alt+Shift+R / Alt+1-9 shortcuts wired. NOW panel's OFFLINE, IDLE, EXITED, and shell-running variants all render correctly with real uptime and PID.

What is wired but mute. TerminalMetaMessage.title in the type declaration — the sidecar never emits it, the pane never reads it. window.__HOME__, which compactPath() reads to produce ~/foo-style paths, is never injected — so every path across the UI renders as a full absolute string. CreateTerminalSessionInput.cwd is never sent from the UI, so all sessions spawn in the sidecar's cwd, not the user's project directory.

The TASK STACK hole

When a claude or opencode session is running, the NOW panel is supposed to show a task stack — what the agent is doing. The code for that branch (TerminalPane.tsx:828) renders exactly one line: "Tasks will land here as the agent works." There is no data source, no hook, no polling, no protocol. The WebSocket carries raw PTY bytes; structured task events from the agent would require either heuristic stdout parsing or a second event channel that does not exist.

Gaps

  1. High TASK STACK is pure placeholder.

    The headline affordance of the NOW panel — agent task introspection — is a static string.

  2. High No sidecar lifecycle management.

    Manual npm run terminal-service. No auto-start, no Electron child-spawn, no WebSocket reconnect. Sidecar death ⇒ page refresh.

  3. Medium __HOME__ and cwd both unset.

    Every path in the UI shows the sidecar's absolute cwd, uncompacted — doubly misleading.

Chapter 04Visual — one living room, three sketches

The Visual family is four surfaces of wildly differing depth, all rolled under /deck/visual and friends. Canvas is the surprise: a real, functional side-panel with Monaco, sandboxed previews, and end-to-end code execution. Comfy is a thin but genuine proxy to a local ComfyUI. Dojo is a static component gallery whose live backend is built and disconnected. The Browser pane is fully plumbed Electron IPC with no caller.

/deck/visual ─► ComfyPane (proxies ComfyUI :8188) /deck/control?tab=studio ─► DojoPane (static specimens, live runner unused) chat sidebar ─► CanvasPanel (monaco + iframe + /api/code/execute) Electron ─► themed-browser ─► BaseWindow(header + page) ── no trigger ──

What ships — Canvas. Multi-tab Monaco, undo/redo with localStorage-persisted revisions, a real execution pipe into lib/tools/code-exec supporting Python/Go/Bash/HTML/React/Three.js with sandboxing, sandboxed iframe previews for bundles, image grid, audio player, and a <model-viewer> 3D tab. This surface is doing work.

What ships — Comfy. A history dashboard that polls /api/comfy/history (proxying http://localhost:8188/history) every 5s; DELETE sends POST /interrupt to ComfyUI; thumbnails render via /api/comfy/view proxy; POST /api/comfy/prompt queues workflows and emits AG-UI events; the "Open UI" button hard-links to :8188. If ComfyUI isn't running, every fetch 502s and the panel shows "No jobs in history."

What is wired but mute — Browser pane. themed-browser.ts is eagerly registered in electron/main.ts before createWindow(). All seven IPC handlers are mounted. preload.ts exposes window.deck.browser with the full API. BrowserHeader.tsx subscribes to browser:state events and wires every nav button. app/browser/page.tsx mounts the header. Nothing in the renderer ever calls window.deck.browser.open(url). The pane is inert until one button invokes one IPC.

What is wired but mute — Dojo. DemoRunner, EventViewer, and DemoCard are built, exported, and the SSE pipeline at /api/dojo/demo + /api/dojo/stream emits the full AG-UI event sequence. DojoPane.tsx never imports DemoRunner. All 8 specimens are driven from hardcoded props.

Three "Send to ComfyUI" buttons

Comfy's preset cards — Stable Audio, Hunyuan 3D, Qwen Edit — are rendered with disabled and title="Preset workflows not yet implemented". The queue endpoint works; the workflow JSON payloads have not been authored. This is the entire gap: three JSON files and three button handlers.

Gaps

  1. High Dojo's live demo backend is orphan code.

    Every specimen shows static prop data even though the SSE pipeline exists. One-line import would wire it.

  2. High Browser pane has no launch trigger.

    A complete Electron sub-window system with zero callers. One button anywhere in the deck invokes it.

  3. Medium Comfy preset workflows are permanent stubs.

    The only user path to queue generative work from within the deck is three disabled buttons.

Chapter 05Audio — two halves, both unfinished differently

The audio surface was the provoking context for this chronicle. Live was just rebuilt from first principles — the 518-LOC engine.ts retired, a diff-reconciled transport.ts swapped in, a four-zone FL-native pane shipped. Voice is older and deeper. Both have holes, different kinds.

Voice

Provider is a local Python sidecar on ws://<host>:8000/ws — not Hume, not OpenAI Realtime. Three TTS engines (piper / xtts / chatterbox); Whisper STT. useVoiceChat captures audio/webm;codecs=opus via MediaRecorder, base64-encodes over WS, gets binary blobs back, runs them through compressor + EQ + 8% wet reverb, and plays via AudioBufferSourceNode. VoiceModeSheet wraps this into a continuous-listen loop with VAD, push-to-talk, barge-in, streaming sentence-by-sentence TTS, and pre-baked filler audio (5 WAVs in /audio/fillers/) to cover LLM latency.

The REST fallback routes exist at /api/voice/stt and /api/voice/tts. useVoiceChat never calls them — it only uses the WebSocket. Any cloud fallback would require rewriting the transport layer.

Live

model.ts Channel · Pattern · PlaylistClip(Pattern|Audio) · Insert · LaunchGroup · Song store.ts SongStore — immutable-by-reference, diffable transport.ts LiveTransport — subscribes store, reconciles Tone.js graph on diff mini.ts tinynotation parser script.ts DSL (BPM, track patterns, FX chains, sample directives) importer.ts script → SongStore (creates pending AudioClips for `sample N:` lines) UI: TransportBar · PatternRack · Playlist · LaunchBar · Mixer — the four-zone FL-native pane

What ships. Tone.getTransport().start/stop, BPM ramp, master gain ramp. Step pattern → Tone.Part scheduling with BBS mapping. AudioClip → Tone.Player sync. The diff-reconciler handles add/remove/change on all four model dimensions without full rebuild. Drum voices (MembraneSynth, NoiseSynth, MetalSynth), synth (PolySynth fatsawtooth), sampler (Tone.Player). Five builtin FX (reverb, delay, chorus, AutoFilter, distortion) with wet + bypass. Scene-launch with quantized triggers (immediate/beat/bar/2bar/4bar) and clean handoff between scenes.

What is wired but mute. AudioClip.generation carries {prompt, duration, loader, status:"pending", jobId?}. importer.ts stamps status:"pending" clips when a sample N: line is parsed. The API route /api/live/sample exists and would call ComfyUI. The AI tool live.generate_sample is in lib/tools/executor.ts:1079. There is zero UI code that calls the API, polls for completion, or writes the artifact URL back to the clip. Pending clips sit in the Playlist forever with "…generating".

What is wired but mute — the silent bug. Insert.gainDb is a model field. The Mixer slider writes it to the store. The store round-trips it. LiveTransport never reads it. InsertNodes uses bare Tone.Gain(1) with no volume control. The per-insert gain fader is cosmetic; it moves nothing.

Honest about Live

The Live rebuild shipped the skeleton and the nervous system: model, store, reconciler, UI layout, scene-launch. It did not ship the flesh: AI generation is dead, insert gain is silently broken, there's no persistence (serializeSong / deserializeSong exist in model.ts:268 with no callers), piano is a placeholder poly-synth, WAM2 is a passthrough, automation doesn't exist, sidechains don't exist. The scene-launch boundary handoff is real and worth being proud of. Everything else is a promise.

Gaps

  1. Critical AudioClip AI generation is a dead end.

    Pending clips never resolve. Full round-trip missing: no UI→API call, no polling, no patch-back to SongStore. The novel feature is a type stub with unreachable machinery behind it.

  2. High Insert gain fader is decorative.

    Insert.gainDb written, never read by the transport. Replace InsertNodes.input with Tone.Channel; wire gainDb in rebuildInsertChain.

  3. High No song persistence.

    Reload ⇒ total loss. serializeSong / deserializeSong exist; nothing calls them.

  4. Critical Voice has no fallback path.

    Python sidecar down ⇒ entire voice surface disabled. REST proxies at /api/voice/* exist but the hook doesn't use them.

  5. High Voice turns are invisible to chat history.

    VoiceModeSheet's conversationRef wipes on close. No write to thread context.

Chapter 06Agent Infrastructure — one functional tab, four signs

The /deck/control plane has five tabs: Runs, Tools, AgentGo, Models, Studio (Dojo). The standalone routes /deck/runs, /deck/models, /deck/tools, /deck/agentgo all redirect into Control. One of these tabs — Runs — is a deeply-functional real panel. The rest are scaffolding of varying intent.

ControlPane (tabbed shell) ├─► RunsPane ── reads SQLite, SSE refresh, GLYPH view, clear [ships] ├─► ToolsPane ── service health + GPU + quick-actions [mislabeled] ├─► AgentGoPane ── needs Go server on :4243 [conditional] ├─► ModelsPane ── Ollama tags only [narrow] └─► DojoPane ── static specimens [scaffold]

Runs. Reads from SQLite via GET /api/agui/runs. Shows token counts, cost, status dots. SSE at /api/agui/stream refreshes live. Clicking a run fetches its events. GLYPH view mode shows tool payload comparison. Clear button deletes all. This is a real observability panel.

Tools — the most misleading label. The tab is not a tool browser. It is a service-health dashboard: GPU stats via nvidia-smi, health of Ollama/ComfyUI/Terminal/VectorDB/SearxNG/Voice, latency, and "Quick Actions" that open external URLs. The 25 tools in lib/tools/definitions.ts — with full Zod schemas, descriptions, parameters — are invisible. No schema browser, no invocation UI, no way to run a tool directly. A user expecting "see and test the tools" finds "see if the servers are up."

Models — narrowed to one provider. lib/llm/providers.ts is production-ready for ten backends (OpenAI, Anthropic, Google, DeepSeek, OpenRouter, Ollama, llama.cpp, vLLM, LM Studio) with slot-based configuration, health checks, model listing, and runtime override. ModelsPane reads only from /api/ollama/tags. A user running GPT-4o or Claude has no signal in this UI.

AgentGo — blocked on one missing endpoint. useAgentGoRun's dual-stream logic and client.startRunWithStream are complete — they call POST /runs/stream and expect X-Run-Id back. The Go server does not implement that endpoint. The pane forces useTextStream: false and falls back to SSE-only. Higher overhead, no backpressure.

Two registries, neither surfaced

The deck has two tool registries that don't know about each other: lib/tools/definitions.ts (25 LLM tools for the chat pipeline) and lib/plugins/registry.ts (5 widget data-sources). They share concept names (web_search appears in both) but have no shared schema, no deduplication, and no UI that shows either — let alone both.

Gaps

  1. Critical "Tools" pane isn't a tool surface.

    25 Zod-typed tools, zero UI for them. Add a catalog component reading TOOL_DEFINITIONS; ideally a "Run tool" form through /api/tools/bridge.

  2. High Models pane hides the multi-provider layer.

    Ten backends supported, one surfaced. No way to see active provider, switch it live, or see cloud health.

  3. Medium AgentGo text streaming is blocked on a missing Go endpoint.

    JS side is done. Go server needs POST /runs/stream.

  4. Medium No run↔model correlation.

    RunRow.model is a string; no cross-link to the Models pane; cost is aggregated globally.

  5. Medium Plugin/tool registry split is invisible.

    Adding a new capability requires knowing which registry applies. There is no unifying surface.

Chapter 07Native Shell — the deepest room, with two gas leaks

The native layer is, surprisingly, the most thoroughly thought-through subsystem in the repository. Electron main starts Next.js as a subprocess, opens a single BrowserWindow, runs a random-port HTTP portal bridge, and owns an xdg-desktop-portal RemoteDesktop D-Bus session for Wayland key/pointer injection. A themed-browser service wraps window.open into a BaseWindow with a themed header strip. wl-activate.py performs the GNOME-specific focus-activation dance via GTK4 + xdg_activation_v1.

Electron main (process) ├─► BrowserWindow ── Next.js at :3333 (dev) / embedded standalone (packaged) ├─► preload ── window.deck.invoke(channel, ...) and window.deck.browser ├─► portal bridge HTTP (secret-auth, Linux only) │ ←── Next.js server process (can't use IPC, uses HTTP) ├─► RemoteDesktopSession ── org.freedesktop.portal.RemoteDesktop │ sendKeyCombo · tapKeysym · typeString · clickPixel ├─► themed-browser ── BaseWindow(40px header + full page view), IPC wired ├─► wl-activator ── shells out to wl-activate.py per call └─► screencast session ── screencast-capture.py / ScreenshotPortal fallback

What ships. The app starts. The portal path for key injection is real: agent issues native_key({key:"Ctrl+l"}) → linuxAtspiAdapter → portal HTTP callPortal({op:"key"}) → sendKeyCombo → NotifyKeyboardKeysym press/release via D-Bus. User grants the portal session once; a restore token lives under userData/portal-restore.token for silent re-init. The themed-browser IPC round-trip is complete. attach-harness.sh probes the (opt-in) CDP port and execs browser-harness with the right env.

What is wired but mute. ipcMain.handle("portal:status", ...) at main.ts:429 is registered. No renderer, no tool executor, no API route calls window.deck.invoke("portal:status"). It is a probe nothing probes. lib/tools/native/input-common.ts is a @nut-tree/nut-js shim with zero importers — the pre-portal fallback that wasn't removed. Darwin and Windows native adapters are explicit unsupportedAdapter stubs with "Phase D.2" in the comment.

The typing bug that will bite you

RemoteDesktopSession.typeString decides shift per-character with ch >= "A" && ch <= "Z". That's the whole shift check. Any typed string containing !, @, (, ), _, +, {, ", :, ? gets the un-shifted keysym injected — producing 1 instead of !, 9 instead of (, - instead of _. The ATSPI fallback uses KEY_STRING and sidesteps this. The portal path — which is preferred when PORTAL_URL is set — does not. Agents typing anything with punctuation will produce silently wrong output.

Gaps

  1. Critical typeString mis-shifts punctuation.

    A shift-table for the US layout (at minimum) is the fix. Currently: most typed strings with punctuation are silently wrong.

  2. Medium Portal bridge discovery is fragile + unobserved.

    Handoff file re-read only on pid mismatch; portal:status IPC has no caller. Failures are invisible to the UI.

  3. Medium ScreenCast has two layers of Python shell-out, no retry.

    Transient Start failed drops the session; ScreenshotPortal fallback sets interactive:true which re-prompts the user on every capture for non-sandboxed processes.

Chapter 08Cross-cutting chrome — the lie of the bars

DeckShell stacks six providers — settings, thread manager, canvas, inspector, preflight, warp — and renders the frame: sidebar, optional thread rail, main, canvas panel, right-rail toggle strip, inspector sheet, command palette, settings drawer. Warp theming writes data-warmth/type/accent/theme on <html>; Tailwind resolves token aliases through CSS vars. PreflightGate probes /api/preflight/status and appends a dismissible banner.

PreflightGate does not gate. The component's comment is honest: "Never block the UI. Always render children." It renders a banner when services are down. The two preexisting TypeScript errors on lines 136 and 139 are a narrowing failure: the showBanner guard above the JSX block has already excluded "probing" from state, so state === "probing" in the Retry button props is literally unreachable. The disabled prop will never activate; the "Checking…" label will never show.

CommandPalette works — mostly. ~20 commands across navigation, settings, and actions. Navigation router.pushes; settings commands flip provider setters; action-clear-runs really does DELETE /api/agui/runs. A "Shortcuts" category of entries has action: () => {} — they render in the palette and do nothing when selected.

Warp theming is functional. Real DOM attributes, real CSS-var resolution, real localStorage persistence under controldeck.warp.v1. The name evokes Warp-terminal but the integration is thematic only. components/Tweaks.tsx is an older parallel implementation (different storage key, controldeck.tweaks.v1) kept as a handoff spec — not mounted.

Chapter 10What it says

Every codebase is a statement. This one is saying, clearly and loudly, "I want a certain shape." The shape is coherent: a single deck with deep rooms for chat, terminal, visual creative, audio creative, agent operations, and native integration — all woven through a common event protocol, a common theming layer, a common command palette, and a common preflight. The shape is good. The shape is defensible. The shape is consistent.

But the shape has been scaffolded far ahead of the flesh, and the deck has not yet paid the cost of that choice. The cost is visible in three places:

1. The mute wire problem. Half the capabilities the code announces are unreachable by a user. portal:status, window.deck.browser.open, DemoRunner, Dojo AG-UI, serializeSong, buildSystemPrompt, setRuntimeProvider, Insert.gainDb, plugin host, CostIncurred, StepStarted, cloud voice fallback — all wired, all mute. Each one is a decision-shaped hole. Someone thought carefully about the type, the endpoint, the handler. Nobody built the one call-site that closes the loop. The bill for scaffolding is always the same size as the scaffolding, and the deck hasn't paid it.

2. The fake data problem. Three UI elements — SystemHealth's CPU/RAM/Disk bars, RecentActivityWidget, three Comfy preset buttons — present fabricated or permanently-disabled state as real. This is a small surface, but its cost is outsized: it teaches users (including future-you) to distrust every widget, because there's no reliable cue to tell the real from the decorative. A single data-real="false" attribute styled with a dashed border would inoculate the whole deck.

3. The mislabel problem. The "Tools" tab is not a tool surface. The "Models" tab is not a provider surface. These names make promises the panels don't keep. The fix is either rename (cheap, dishonest) or build (honest, slow). The honest fix is one week of work on each to turn them into what their names say they are.

The real architectural insight

Control Deck is not incomplete. It is overbuilt at the protocol layer and underbuilt at the call layer. Every event type, every IPC handler, every hook, every reducer, every tool schema is in place. The missing work is almost entirely one-line caller insertions that close each protocol into a user-reachable loop. The deck does not need more architecture. It needs a focused week per surface closing the loops that are already open.
