Briefing 03 · Product-aligned roadmap

Two weeks to make Control Deck more trainable without collapsing it into “just a harness.”

This roadmap treats Control Deck as a full product first: UI shell, runtime, control plane, machine interface, artifact system, and multimodal workstation. The training-facing work exists to make that product legible, replayable, and improvable by external training stacks.

Rule 1: do not flatten the product Rule 2: export traces, don’t embed the trainer Rule 3: ship measured regressions, not vibes

Roadmap thesis

The point of this sprint is not to turn Control Deck into a training framework. The point is to make the real product observable enough, stable enough, and exportable enough that models can be trained against it without dumbing it down.

Protect

Keep the full product identity: chat, runs, models, hardware, native actions, multimodal outputs, operator control.

Instrument

Make observations, actions, approvals, artifacts, and outcomes explicit and replayable.

Export

Feed external SFT / preference / RL stacks with clean trajectories instead of stuffing training code into the app.

Three sprint tracks

Track Purpose Examples
A · Product integrity Preserve correct behavior of approvals, runs, panes, artifacts, and model routing. Approval naming fix, route tests, run correctness, operator-visible debug state.
B · Environment legibility Make the deck understandable to both humans and policies. Observation schema, capability registry, replay bundles, deterministic task packs.
C · Training handoff Export trajectories and labels cleanly to external training systems. SFT trace export, preference pair generation, reward rows, eval fixtures.

Day-by-day plan

Day 1 · Canonical action semantics

Goal: one stable action vocabulary across bridge, approvals, live tools, and run logs.

  • Normalize live tool naming and approval interpretation.
  • Write regression tests for canonical action names.
  • Document the canonical action namespace.

Day 2 · Contract tests for the product backbone

Goal: protect the highest-risk product contracts, not just the harness parts.

  • Add route-level tests for chat, tools bridge, glyph-eval, inference providers.
  • Test malformed args, unsupported tools, missing approval, and failure envelopes.
  • Keep tests close to operator-visible behavior.

Day 3 · Commit the in-repo environment probes

Goal: move the GLYPH/model/tool matrix out of ad-hoc scripts and into the product repo.

  • Commit the matrix harness.
  • Make model/provider inputs configurable.
  • Persist machine-readable output artifacts.

Day 4 · Observation schema v0

Goal: define what a model “sees” when driving Control Deck.

  • Capture thread context, run phase, tool manifest, approval state, provider state, hardware snapshot, artifact handles.
  • Keep it compact enough for replay and training export.
  • Do not reduce the deck to only messages + tool calls.

Day 5 · Run export bundle

Goal: every meaningful run can become a portable training/eval object.

  • Bundle run metadata, ordered events, approvals, artifacts, tool invocations, model/provider context.
  • Version the bundle schema.
  • Export at least one real run end to end.

Day 6 · Replay at the contract level

Goal: prove Control Deck can replay its own behavior without requiring full token determinism.

  • Compare expected vs observed tool/action sequence.
  • Track approval compliance, payload shape, and outcome state.
  • Make pass/fail machine-readable.

Day 7 · Capability registry

Goal: record measured local behavior per model, not only curated benchmark metadata.

  • Store tool JSON validity, GLYPH parse rate, approval compliance, recovery behavior, latency, quirks.
  • Persist per model/provider/as-of timestamp.
  • Render at least one real model’s local scorecard.

Day 8 · Product-native task pack v0

Goal: define tasks that reflect Control Deck as it really exists.

  • Chat/tool task
  • Approval-sensitive task
  • Model-selection or provider-selection task
  • Run/artifact inspection task

These should feel like product workflows, not synthetic benchmark trivia.

Day 9 · Dataset export modes

Goal: turn product traces into training-ready records.

  • Successful runs → demonstration/SFT examples
  • Human revisions or preferred outcomes → preference pairs
  • Replay outcomes and heuristics → reward/eval rows
  • Failed tool or approval paths → recovery datasets

Day 10 · Critical gate

Goal: block regressions in behavior that matters to the real product.

  • Approval compliance
  • GLYPH/tool parsing
  • One replay case
  • One product-native task case
  • One failure-recovery case

Day 11 · Operator-visible debug views

Goal: let a human understand why a model failed to drive the deck correctly.

  • Readable expected vs actual action diff
  • Approval timeline
  • Artifact and run context links
  • Compact failure summary per replay/task

Day 12 · Expand to richer surfaces

Goal: start respecting the fact that Control Deck is more than chat and tools.

  • Add at least one native/browser/workspace task.
  • Add at least one multimodal or media-oriented task if the local setup supports it.
  • Mark these as higher-variance but product-critical.

Day 13 · External training handoff spec

Goal: cleanly separate Control Deck from the trainer while making the bridge obvious.

  • Document export schema for SFT / preference / reward pipelines.
  • Show how an external stack like TRL/Axolotl/Atropos would consume it.
  • No weight updates inside the app.

Day 14 · Hardening and scorecard

Goal: end with evidence, not aspirations.

  • Run the critical gate.
  • List what shipped, what failed, what was cut.
  • Define the next 3 moves based on measured pain, not speculation.

What this roadmap protects that a harness-only plan would miss

It protects product identity

  • The model is trained against approvals, artifacts, model routing, and operator surfaces.
  • The deck remains a product with real workflows.
  • UI, native reach, and multimodality remain first-class.

It still gives you the training substrate

  • Stable action schema
  • Observation schema
  • Replay bundles
  • Capability metrics
  • Dataset exports
  • Regression gates

Scope cuts if time slips

Cut first Keep no matter what
Fancy UI polish for scorecards, broad route coverage beyond core contracts, ambitious multimodal task packs, generalized benchmark ingestion. Canonical action semantics, approval correctness, export bundle, replay path, capability registry, critical gate, and at least one product-native task pack.

End-state statement

At the end of this sprint, Control Deck should be able to say: “We can capture, replay, score, and export real product behavior well enough to train models that learn to drive the deck — without shrinking the deck into a toy environment.”
This roadmap intentionally keeps training infrastructure external and treats Control Deck as the product and environment that produces the traces, labels, and eval cases.
