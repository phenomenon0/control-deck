Briefing 02 · Subsystem map

If you train models to drive Control Deck, you are training against a layered environment.

The repo shape makes this clear: a policy is not learning one API. It is learning how to move through a stack of observations, actions, approvals, artifacts, hardware signals, and modality-specific surfaces.

High-level architecture map

Human operator
  ↓
Deck UI shell (app/deck/* + components/panes/*)
  ↓
Conversation + activity surfaces (chat, runs, approvals, artifacts)
  ↓
API/control plane (app/api/*)
  ↓
Execution backends (tools, inference, native OS, terminal, workspace, media)
  ↓
Persistence + state (SQLite / AGUI events / settings / artifacts / invocations)
  ↓
External systems (models, browser targets, OS windows, files, GPUs, media services)

Training models to “drive Control Deck” means deciding what observation slices and action channels from this stack should become policy inputs and outputs.

Subsystem inventory and training relevance

Subsystem Repo signals What it does in the product Why it matters for training
UI shell and panes app/deck/*, components/panes/* Exposes the operator-facing surfaces: chat, models, hardware, runs, settings, tools, voice, live, visual, workspace, terminal. This is the observation surface. Even if training starts with tool traces rather than pixels, the model needs a coherent view of what surface it is operating inside.
Chat + AGUI + runs app/api/chat/*, app/api/agui/*, lib/agui/*, runs panes Handles streaming, run events, approvals, threads, artifacts, and message history. This is the trajectory backbone. It should become the canonical source for demonstrations, replay bundles, failure labels, and evaluation traces.
Tool runtime + bridge + approvals lib/tools/*, app/api/tools/bridge/route.ts, lib/approvals/gate.ts, lib/codec/* Defines the typed action space, argument schemas, GLYPH transport, execution routing, and approval decisions. This is the most important training substrate early on. If the action space is unstable, everything downstream gets noisy.
Inference control plane lib/inference/*, app/api/inference/*, model/hardware panes Routes text, image, video, audio, vision, embeddings, rerank, provider metrics, and benchmark metadata. Lets you train policies that make model-aware and modality-aware choices instead of blindly calling a single default model.
Hardware awareness lib/hardware/*, app/api/hardware/*, app/deck/hardware Reads GPU/VRAM/process state, provider availability, offline models, and machine profile. Enables resource-aware decision making: pick a local model, avoid VRAM contention, or explain why a task should route elsewhere.
Native/browser/workspace execution lib/tools/native/*, terminal/workspace libs, Electron/browser wiring Gives the system reach into the OS, browser-ish targets, shells, and code sandboxes. This is where Control Deck becomes more than an LLM dashboard. It is the environment layer for genuinely agentic training tasks.
Multimodal creative surfaces voice, comfy, live, audio, visual, widgets Supports media generation, live transport, voice I/O, and richer artifact workflows. High-value later-stage curriculum. These surfaces let you train multimodal workflows instead of purely textual tool use.
Plugins, skills, rules lib/plugins/*, lib/skills/*, lib/rules/*, related API routes Adds extensibility, procedural memory, and declarative policy surfaces. Useful for advanced training: models can learn when to invoke structured capability packs, not only raw tools.

Training priority tiers

Tier 1 · Essential first

  • Tool runtime and schema-valid action emission
  • Approvals and side-effect discipline
  • AGUI runs/events/artifacts as trace backbone
  • Replayable chat/tool trajectories
  • GLYPH/tool parsing reliability

This is the smallest stable loop for SFT/eval/reward collection.

Tier 2 · Product-native depth

  • Inference routing decisions
  • Hardware-aware task planning
  • Run inspection and artifact interpretation
  • Model/provider selection under constraints

This is where the model starts feeling like it understands the deck.

Tier 3 · Rich environment control

  • Native OS automation
  • Browser-target workflows
  • Workspace and terminal multi-step tasks
  • Voice/media/live/Comfy multimodal flows

This is the highest-value frontier, but it needs the earlier tiers to be stable.

What to export from each subsystem

Observation exports

  • Thread state and recent messages
  • Run phase and current tool status
  • Available tool manifest + schemas
  • Approval mode and pending approval queue
  • Provider/model availability and current routing config
  • Hardware snapshot: VRAM, active model processes, offline availability
  • Artifact handles and compact previews

Action exports

  • Typed tool calls with arguments
  • Approval request / approve / reject / revise flows
  • Model or provider switching
  • Workspace/terminal/browser/native actions
  • Stop, retry, resume, handoff, or ask-for-clarification acts
  • Artifact-producing actions and post-hoc evaluation markers

The crucial point: the training interface should be a faithful projection of the real product, not a toy abstraction that strips away approvals, hardware state, artifacts, or modality choices.

Best near-term training loop

Real Control Deck task
  → observation snapshot
  → policy action(s)
  → tool/native/browser execution
  → approvals / interruptions / artifacts
  → run + event persistence
  → export trajectory
  → convert to demos / preference pairs / reward rows / eval cases
  → train in external stack
  → replay in Control Deck

This keeps Control Deck true to itself while still making it the center of model improvement.
