Prologue

One cockpit, many surfaces

Control Deck is what you get when you stop treating an AI assistant as a chat window and start treating it as an operator that needs typed access to every surface on your machine. Not a wrapper around someone else's SDK. Not a chatbot UI bolted onto an API key. A real runtime: an agent process, an inference router, a voice service, a native-OS adapter, and a single shell that ties them together — with one approval gate sitting in the middle so the operator never moves without you seeing the move.

This document is the architect's view. It assumes you already know what the deck does at the surface level — chat, tool calls, code execution, native automation, voice — and answers the deeper question: what are the structural decisions that made it look this way, and where are the seams. Every chapter ends in opinions, not just descriptions. Extend, cut, keep, or watch — explicit verdicts on each piece.

The substrate is more interesting than the surface. Tools are how an agent does anything, and Control Deck spends most of its complexity on tools. Get the substrate right and the surface can be rebuilt in a weekend.

The deck weighs in at roughly 5,700 LOC for the five files that matter most — lib/agui/db.ts, lib/tools/executor.ts, lib/tools/definitions.ts, components/chat/ChatSurface.tsx, and lib/hooks/useAgentRun.ts. Around them sit ~40 typed tools, three native-OS adapters, a Python voice service, a Next.js standalone server embedded in Electron, and a Streamable HTTP MCP endpoint that lets other agents drive the deck the same way the deck's own agent does.

The system has a centre of mass: events flow through one bus, schemas define every boundary, and every side-effect goes through one gate. Everything else is a consumer or a producer sitting on those three rails. The rest of this manual is the consequence of those three rules.

Chapter 1

Origins: the four pressures

Most architecture documents describe what's there. Few describe the forces that shaped it. Control Deck's shape comes from four pressures pulling at the same time:

Pressure 1
The agent must drive real surfaces.
A chat-only agent is a toy. A real cockpit lets the agent open a browser tab, type into a native window, run code, search vectors, render an image, drop a file. Each of those is a different API, a different latency profile, a different failure mode. They have to look the same to the agent.
Pressure 2
The user must see every move before it lands.
An agent that runs bash or execute_code without confirmation is a footgun. Approval has to be centralized, not per-tool — one place to audit, one place to add rules.
Pressure 3
The agent process is its own thing.
LLM context state, tool-loop scheduling, mid-run cancellation, partial-result streaming — those don't belong in a Next.js API route. Agent-ts runs separately and emits a typed event protocol. The UI is just one consumer.
Pressure 4
The deck is itself an MCP backend.
If the deck has 35 useful tools, other agents — Claude Code, Cursor, Cline — should be able to use them too. Same gate, same executor, different transport. This is the property that justifies the schema-first design.

Pull these four ways and you get the system you have: a typed tool runtime, a centralized gate, a separated agent process speaking a typed event protocol, and an MCP server that re-exposes most of the tool surface to external agents. Everything else — Canvas, voice, the workspace, the per-OS native adapters — is downstream of those four.

Design Principle

Schemas, not APIs, are the contract. Every tool is a Zod schema. The schema generates the LLM-visible manifest, the bridge validator, the TypeScript type, the MCP tool definition, the catalog row, and the gate's side-effect lookup. Adding a tool means writing a schema, an executor case, and three allowlist entries. Forty tools later, that pattern still scales.

Chapter 2

Process topology

Three or four processes, depending on configuration. Electron supervises them; start-full-stack.sh is the canonical dev launcher. Each process has a single responsibility and one well-defined surface to its neighbours.

Figure 2.1 — Process topology. Electron supervises three children. Voice-core is lazy.

Why these boundaries

The Next.js process owns persistence and the UI. Agent-ts owns the LLM loop. Voice-core owns the audio pipeline. Each boundary is a deliberate seam: the agent loop is async, long-lived, and OS-agnostic, so it runs in its own Node process — restarting it doesn't kill your UI session. The voice pipeline is Python because the best STT/TTS libraries are Python-first, and it's a separate process so Python crashes don't take down the deck.

Tool execution lives in the Next.js process, not in agent-ts. That's the most surprising boundary — and the most load-bearing. Agent-ts requests a tool call via HTTP to POST /api/tools/bridge; the Next.js process runs the tool, hits the database, manages approvals, emits events into the event bus. This puts tool execution next to persistence and the UI, which matters because:

  • Approval state lives in SQLite — tools that need approval need DB access.
  • Artifacts get URLs (/api/artifacts/<runId>/<filename>) — tools that emit artifacts need to write to disk under a Next-routed directory.
  • The same executor serves the in-app agent and external MCP clients — putting it in agent-ts would force the MCP server to call back out to the agent process, which would deadlock during a long tool call.
Keep
Tool execution in the Next.js process. Putting tools next to persistence + the MCP server is the right choice. Moving them to agent-ts would create a circular HTTP dance for every approval and every artifact. Resist the temptation.
Watch
The voice-core process boundary. Splitting voice into its own service was reasonable when STT/TTS providers were heterogeneous and Python-first. With more modern JS-friendly endpoints (cloud TTS over HTTP, browser-native STT) the case for a separate process weakens. If voice-core becomes a thin proxy, fold it in.
Extend
The Electron supervisor. Right now it's mostly a launcher. It could be a real supervisor: restart-on-crash for each child with backoff, structured log multiplexing, a single health-check endpoint. The bones are in electron/services/*-supervisor.ts — finish that work, it pays for itself on the first agent-ts crash.
Chapter 3

Layered architecture

Inside any single process, requests fall through a stack of layers, each one validated by a Zod schema at its top edge. This isn't an arbitrary OO design — it's the natural shape when you commit to "schemas are the contract." Crossing a layer means proving you have the right shape.

Figure 3.1 — Layered architecture. Every edge between layers is a Zod-validated boundary.

Reading from top to bottom: the surface dispatches an action, the state hook updates, the wire layer carries an SSE stream of AG-UI events, the agent loop emits those events as it works, the tool layer dispatches a specific implementation, the gate decides whether to run it, and the effect layer makes the actual change to the world. Reading bottom to top: an effect emits a result, the gate stamps it approved, the tool layer wraps it in a result envelope, the agent loop turns it into an event, the wire layer streams it back, the state hook reduces it into segments, the surface renders.

Two arrows are doing real work here. The down-arrow is action — what the user wants. The up-arrow is observation — what actually happened. They never cross without a schema gate in between.

Keep
The layered model. It's the single largest reason this codebase remains comprehensible at its size. Every new feature gets one obvious home in the stack.
Chapter 4

Tool runtime: schemas as contract

The tool runtime is the most distinctive piece of architecture in Control Deck. It is also the riskiest to change. Forty tools live in one Zod discriminated union. One executor switch dispatches all of them. Three allowlists determine which tools are visible to which clients.

The schema layer

Every tool starts in lib/tools/definitions.ts as a Zod object with two required fields: a literal name and an args object. The discriminator is the name. Adding a tool is a schema:

const ExecuteCodeSchema = z.object({
  name: z.literal("execute_code"),
  args: z.object({
    language: z.enum(["python", "lua", "go", "c",
                      "javascript", "typescript", "bash", "sh",
                      "html", "react", "threejs"]),
    code: z.string(),
    filename: z.string().optional(),
    args: z.array(z.string()).optional(),
    stdin: z.string().optional(),
    timeout: z.number().int().min(1000).max(60000).optional(),
  }),
});

From this one schema, six artefacts fall out:

Manifest
The JSON-schema the LLM sees. Zod → JSON-schema → tool calling spec.
Validator
The bridge endpoint validates args with the same Zod object before dispatching.
Type
type ExecuteCodeArgs = z.infer<...>. The executor case sees a real TS type.
MCP definition
Re-exported as an MCP Tool at /api/mcp. Same schema, different transport.
Catalog row
GET /api/tools/catalog serves risk + sideEffect + timeoutMs.
Gate lookup
The side-effect set in loop.ts and gate.ts uses the same names.

The dispatch layer

lib/tools/executor.ts is a forty-case switch. There is no plugin loader, no registry table, no factory pattern. The argument has been made for fancier dispatch — the team has resisted, and they're right. At forty tools, a switch is the most readable thing in the room.

The executor dispatch (excerpt) 
switch (tool.name) {
  case "execute_code":
    return await executeCodeTool(tool.args, ctx);
  case "generate_image":
    return await executeGenerateImage(tool.args, ctx);
  case "native_locate":
    return await executeNativeLocate(tool.args);
  case "vector_search":
    return await executeVectorSearch(tool.args);
  // ... 36 more cases ...
  default:
    return {
      success: false,
      message: `Unknown tool: ${tool.name}`,
      error: "Unknown tool",
    };
}

The result envelope

Every executor returns a ToolExecutionResult with the same shape:

{
  success: boolean,
  message?: string,
  data?: Record<string, unknown>,
  artifacts?: Array<{ id, url, name, mimeType }>,
  error?: string,
}

The result then gets wrapped twice: once into a DeckPayload envelope ({kind: "json", data: ...}) and again into an AG-UI ToolCallResult event. The double wrapping is a real source of confusion — it caused the bug fixed in commit 5edf81c, where the chat UI couldn't find code and language because it lived at result.data.details.data.

Aha moment

The double-wrap exists because DeckPayload was added after the executor was already returning shaped results. Rather than rewriting forty executors, the wrap was added around them. The cost is one nested lookup; the benefit is that DeckPayload can carry glyph or text variants without disturbing JSON tools.

The three allowlists

This is where the design loses elegance. Three lists must stay in sync:

ListFilePurpose
BRIDGE_TOOL_LISTbridgeToolList.tsWhat agent-ts can call through the bridge.
TOOL_MANIFESTmanifest.tsWhat the in-app agent sees.
MCP_PROFILESmcpProfiles.tsWhat external MCP clients see (per profile).

These three are not derived from each other. They're hand-maintained. If you add a tool and forget to update one of them, the tool exists but is invisible to that surface. There's no compile-time check.

Keep
The Zod-first design. The "schema generates everything" approach is the keystone. Don't introduce a separate manifest format. Don't move to a runtime registry. The compile-time exhaustiveness of the discriminated union is doing more work than it looks like.
Watch
The 1305-line executor.ts. At forty tools, the switch is fine. At a hundred it won't be. The fix isn't a plugin loader — it's splitting the switch by category: executeMedia(args), executeNative(args), executeWorkspace(args). Same shape, smaller files.
Extend
Make the allowlists derived. Annotate each Zod schema with .meta({ surfaces: ["agent", "mcp", "bridge"], risk: "high" }). Generate the three allowlists at build time. Compile-time error on drift. Two days of work, removes a permanent category of "added a tool, forgot the list" bugs.
Chapter 5

AG-UI: events end-to-end

The deck speaks a custom event protocol called AG-UI (Agent-to-UI), defined in lib/agui/events.ts. It's a tagged union of event types, each one a Zod schema. Three properties matter:

  • Persisted. Every event written to events in SQLite. Replay is possible.
  • Streamed. Events broadcast via an in-memory pub/sub hub keyed by threadId.
  • Schema-versioned. Each event carries schemaVersion: 2 for forward compatibility.

The event taxonomy

Figure 5.1 — Event taxonomy. All events flow through one in-memory hub, keyed by threadId.

The DeckPayload envelope

Tool-call results and large payloads carry a tagged envelope so the wire layer can transmit different shapes uniformly:

type DeckPayload =
  | { kind: "json";  data: unknown }
  | { kind: "glyph"; glyph: string; approxBytes: number }
  | { kind: "text";  text: string };

The kind: "glyph" variant is interesting — for tools like glyph_motif that emit SVG, the wire carries the SVG string directly plus a size hint so the UI can decide whether to inline or fetch.

Persistence is event-sourced

Every event lands in the events table as a JSON row. A projection task reads those events and builds denormalized rows in messages (with metadata.toolCalls), artifacts, and approvals. The events table is the source of truth — the denormalized tables are the cache.

This pays off in two ways: replay (you can re-derive any thread from scratch) and debugging (every wire event is queryable post-hoc). The sqlite3 data/deck.db command was load-bearing in fixing the recent execute_code bug — we read the raw event to figure out where code actually lived on the wire.

Keep
The events-as-source-of-truth model. The amount of debugging it has saved is hard to quantify. Don't drop it for "performance" without first measuring it actually being slow.
Watch
The optional ToolCallArgs. Today, agent-ts sometimes emits it, sometimes doesn't. The reducer back-fills from the result. This works but is fragile — any new "args carrier" path will need to know the back-fill convention. Either make ToolCallArgs required, or document the back-fill rule in lib/agui/events.ts next to the type.
Chapter 6

The approval gate

Approval is the spine. Every side-effect-y tool call passes through lib/approvals/gate.ts before executor.ts runs it. The gate writes a row to the approvals table, emits an ApprovalRequest event, and blocks until a UI decision arrives.

Figure 6.1 — Approval lifecycle. Every side-effect-y tool routes through one DB-backed gate.

The risk policy

Two sets, defined in apps/agent-ts/src/server/loop.ts:

const SIDE_EFFECT_TOOLS = new Set([
  "bash", "write_file", "edit_file", "execute_code",
  "vector_ingest", "vector_store",
  "native_click", "native_type", "native_key",
  "native_screen_grab", "native_focus_window",
  "native_click_pixel", "native_focus",
]);

const HIGH_RISK_TOOLS = new Set(["bash", "execute_code"]);

bash and execute_code are high. Native input tools (click/type/key/focus) are medium. Read-only tools (locate/tree/screen_grab) are not gated. The UI uses risk level to choose the dialog flavour — a small chip for medium, a full diff modal for high.

The drift problem

The side-effect set lives in two files: loop.ts (the agent process) and lib/approvals/gate.ts (the deck process). They must agree, because each is consulted by a different code path. Today they're hand-maintained. If they drift, you get one of two failure modes: a tool runs without approval (bad) or a tool blocks forever waiting on an approval that was never issued (worse).

The seam to seal

This duplication is the single most fragile piece of the approval system. The two lists exist because they run in separate processes and can't import from each other. The fix isn't deduplication — it's derivation. Annotate each schema with .meta({ risk: "high", sideEffect: true }), generate both lists at build, ship them as static JSON to both processes. Today's fix is "remember to update both"; tomorrow's fix is "the schema is the truth."

Reversibility, not just dangerousness

The risk classification deserves a second pass. Right now it conflates two distinct properties:

  • Dangerousness — could this tool do harm? bash can rm -rf; vector_store cannot.
  • Reversibility — could you undo it? vector_store can be deleted from the collection; native_click on a "Send Email" button cannot.

Today's policy treats these the same. A more useful taxonomy:

AxisLowMediumHigh
dangerreadlocal writeexternal action
reversibilitytrivialmanualnone
blast radiusone rowone appsystem-wide

Two-axis policy lets the gate distinguish "execute_code running in a sandbox" (high danger, high reversibility) from "native_click on a 'Send' button" (medium danger, no reversibility). The first should be a chip; the second should be a modal even though current policy treats them as the same medium tier.

Extend
Move risk policy onto the schema. Tag each Zod schema with .meta({ danger, reversibility, blastRadius }). Derive both SIDE_EFFECT_TOOLS and HIGH_RISK_TOOLS at build time. The deck and agent-ts both consume the same generated JSON.
Extend
Add session-scoped trust. Today every approval is per-call. For a power user running 30 native_clicks in a row, this is friction. Allow "approve all native input for next 60 s" with a clear visual indicator. Audit log still gets one row per call.
Chapter 7

The tool catalog

Forty tools, six categories. The cardinality alone tells you what the project bets on: nineteen native-automation tools is a big bet on driving the host OS, six workspace tools is a smaller but deliberate bet on letting the agent rearrange the UI.

Figure 7.1 — Tool category distribution. Nineteen native tools is a deliberate, expensive bet.

What's missing

The catalog has obvious holes:

  • No read_file. execute_code is the workaround. A real read_file tool (zero side-effects, no sandbox cost) would be cheaper and more useful for code-aware tasks.
  • No HTTP fetcher. web_search does discovery; there's no "fetch this specific URL and return the text" tool. Agents currently use execute_code with Python requests, which is a hammer.
  • No git tool. Surprising for a developer-facing cockpit. Agents fall back to bash.
  • No diff/patch tool. edit_file is the closest, but a real apply_patch would let the agent propose changes the user can accept piecemeal.
Extend
Add read_file, http_fetch, git, apply_patch. Each is a half-day. Each removes a class of awkward execute_code uses. read_file in particular dramatically reduces approval friction.
Cut
The Windows-only native tools while they remain stubbed. native_invoke, native_wait_for, native_element_from_point, native_read_text, native_with_cache, the four native_watch_* tools and native_baseline_* all currently return unsupported_platform on Linux/macOS. They clutter the manifest the LLM sees with options that will never work for the current user base. Either implement them or hide them per-platform.
Watch
The workspace_* tools. Letting the agent rearrange your UI is bold. It's the most magical feature when it works and the most disorienting when it misfires. Keep the synchronous-with-timeout discipline on workspace_pane_call — fire-and-forget rearrangement bugs are nearly impossible to debug after the fact.
Chapter 8

Native adapters: three OSes, one interface

The native automation layer is the most ambitious piece of Control Deck. The interface is uniform — NativeAdapter defines locate, tree, click, type, key, focus, screen_grab. The implementations are radically different per OS.

Figure 8.1 — Native adapters. Linux is the daily-driver target; Windows is the design target.

Why the Python helper on Linux

The Linux adapter doesn't talk to pyatspi from Node directly. It spawns scripts/atspi-helper.py as a long-lived stdio daemon and exchanges JSON-line requests over its stdin/stdout. This pattern keeps the Node side dependency-free while letting the Python side use the rich pyatspi bindings. Same pattern as voice-core, just smaller scope and in-process supervision.

The trick that makes screen-grabs usable: the portal's restore_token is persisted to app.getPath("userData") after the first accept. Subsequent calls reuse it and return a PNG in ~250 ms with no dialog. Without that trick, every native_screen_grab would pop the chooser, which would make the tool useless for agent loops.

Why Windows is stubbed

Read the schema list and you'll see the Windows team had ambition: native_invoke dispatches UIA control patterns (Invoke, Toggle, ExpandCollapse, Scroll, RangeValue, Value, Selection), native_wait_for subscribes to UIA structure-change events (non-polling popup detection), native_with_cache batches sub-ops against a cached subtree for 10-100× speedups on Explorer/Outlook trees, native_watch_install registers background watchers for dialogs. The design is real. The implementation isn't.

Cut
Hide the Windows-only tools on non-Windows. Today they show up in every catalog regardless of platform. They confuse the LLM ("which native_click variant should I use?") and they trip mcp-tool-eval. Filter them at manifest.ts based on process.platform. They're still there when running on Windows.
Extend
Add Wayland-native ydotool fallback for Linux click/type. The current Linux adapter relies on AT-SPI for actions and falls back to xdotool. On modern Wayland sessions, xdotool fails silently for non-AT-SPI widgets. native_click_pixel via RemoteDesktop portal covers some of this, but for keypresses there's no equivalent. The ydotool daemon (running with uinput permissions) closes the gap.
Watch
The handle TTL. Native handles cache for 60 s at /tmp/control-deck-atspi-<uid>.json. Agents must re-locate before acting. This works for short loops but bites in long-running multi-step plans. Consider a "refresh on miss" pattern so the agent doesn't have to manage TTL itself.
Chapter 9

Voice pipeline: a parallel architecture

Voice is the part of the system that earns the strongest opinion. The pipeline itself is excellent — phrase-conductor for sub-sentence chunking, streaming-tts for low-latency playback, a real FSM in session-machine.ts, a cross-surface activity bus. The architectural problem isn't what voice does. It's that voice has its own copy of state that the rest of the deck doesn't share.

Figure 9.1 — The voice/chat split. Same problem solved twice, by accident not design.

The split, concretely

ConcernChat pathVoice path
Conversation arrayuseThreadManager.messagesconversationRef (in-hook)

SSE parseruseAgentRunown parser in hook
Timeline displayChatTimeline (rich)VoiceTranscript (text-only)

Type a message, then speak. The voice path's conversationRef has never seen what you typed. The assistant has effectively forgotten the typed turn. This isn't a bug — it's the architecture.

The unification path

A working plan lives in ~/.claude/plans/hazy-kindling-ladybug.md. Three steps, each shippable on its own:

  1. Remove the parallel conversation. Voice's STT-final dispatches to the chat's normal submit path. session.runTurn, the in-hook fetcher, and conversationRef get deleted.
  2. Move streaming TTS to consume the chat stream. The phrase-conductor + StreamingTtsClient pipeline moves into a ChatSurface effect that runs when voiceActive && isRunning. Voice's own SSE parser is no longer needed.
  3. Replace the fullscreen overlay with an inline indicator. The chat stays visible; voice contributes an orb + partial transcript strip near the composer.
The seam where the bugs live

Every recent voice bug — TTS truncation, stale re-fire, dedup races, "audio button hoards text" — lives in the seam between these two paths. Each was patched with a guard. Step 1 of the plan removes the seam entirely. The patches stay (they protect the streaming-TTS bridge), but the class of bugs that motivated them disappears.

Extend
Execute steps 1–3 of the voice unification plan. The voice path's plumbing — phrase-conductor, streaming-tts, audio-output — is good. It just has the wrong caller. Move the caller from use-voice-session up into ChatSurface and the architecture clicks into place. This is the single highest-leverage change available right now.
Cut
The fullscreen voice overlay. Voice is an input modality, not a separate app. The overlay hides the chat the user is having. After step 3 of the plan, it's gone.
Keep
The activity-bus. Cross-surface coordination (dock mic vs composer mic vs sidecar surfaces) needs a real bus. It works. Don't simplify it away during the unification — it's load-bearing.
Chapter 10

Surfaces: Canvas and Workspace

Control Deck has two parallel surface roots. /deck is the chat-first layout — DeckShell, a CSS grid with sidebar, threads, chat, and (optionally) Canvas. /deck/workspace is the dockview-based multi-pane shell where the agent can rearrange your panes mid-conversation.

Figure 10.1 — Two surfaces. Same panes, different composition rules.

Canvas: tool output as a first-class surface

Canvas is for tool outputs: code, images, audio, 3D models, HTML previews, generated documents. State lives in lib/canvas/; the React store is in lib/hooks/useCanvas.tsx. Two modes:

  • Docked (default) — a grid column to the right of the chat, sized by --canvas-width CSS variable. Resizable, persists to localStorage key deck:canvas:v2.
  • Floating — a position: fixed window with drag from title bar and corner-grip resize. Persisted position survives reloads.

Canvas opens on two triggers: ArtifactCreated events during a live run (auto-opens preview tab), and explicit openCanvas(...) calls from the chat (Play button on a code block, or workspace_show_canvas tool).

Workspace: capabilities, not props

The workspace pane system has the most interesting design in the deck. Each pane declares capabilities — named operations the agent can dispatch synchronously. chat.append_text, terminal.send_keys, canvas.load_code, notes.replace_text, browser.navigate. The agent calls workspace_list_panes to discover what's available, then workspace_pane_call to invoke a capability.

This pattern decouples the agent from pane internals. A new pane type can ship without changes to the agent's prompt — the capability list is discovered at runtime.

Keep
The capability pattern. It's the cleanest way to expose pane operations to an agent. Don't shortcut it with direct DOM access or "pass pane ID into the agent."
Watch
Two surface roots. Maintaining /deck and /deck/workspace in parallel risks divergence. If /deck is the simple/default and /deck/workspace is the power surface, that's fine — but be explicit about it. If they're meant to converge, decide who absorbs whom.
Extend
Canvas state to the DB. Today Canvas tabs live in localStorage. When the agent calls workspace_show_canvas on a different browser window, the tab doesn't appear there. Move Canvas state to per-thread DB rows; sync across surfaces.
Chapter 11

Persistence: SQLite as cockpit memory

One data/deck.db file, 17 tables, opened via better-sqlite3. Synchronous calls on the Node main thread — fast enough that async would just add latency. The 1320-LOC lib/agui/db.ts is the single point of database access.

The two-write pattern

Every tool call writes twice. Once to events as the raw AG-UI event, once to a denormalized table (messages.metadata.toolCalls, artifacts, or approvals). The events table is the source of truth; the denormalized rows are the fast-path projection.

Figure 11.1 — The two-write pattern. Event-sourced with a denormalized cache.

The truncation seam

The denormalization layer has its own trap: messages.metadata.toolCalls stores a compact summary of each step, and the projection truncates string fields at 200 characters. That worked fine until execute_code's code field needed to be preserved for reload. The fix was a carve-out (commit 5edf81c). The deeper lesson: the projection layer should be schema-aware, not blanket-truncating.

Keep
SQLite + synchronous better-sqlite3. For a local-first cockpit this is exactly right. Don't reach for Postgres until you have a real reason.
Extend
Make the projection schema-aware. Each tool's persistence shape should be a separate Zod schema (e.g. ExecuteCodeSummary, NativeClickSummary) deriving from the args schema. The current "truncate everything to 200 chars" pattern will keep biting until it's replaced.
Watch
The voice_* tables. Five tables for a feature that the architecture is about to fold into the chat path. If the unification plan lands, several of these will be dead weight.
Chapter 12

The deck as MCP backend

Control Deck is the only system I've seen that uses MCP both ways. It hosts an in-app agent that calls tools, and it exposes its tools to external agents via a Streamable HTTP MCP endpoint at POST /api/mcp. 35 of the 40 tools are surfaced (web_search and a handful of internal live.* tools are excluded).

External callers — Claude Code, Cursor, Cline, or anything speaking MCP — register the deck and get tools that look indistinguishable from local ones. The dispatch hits the same executor, the same approval gate, the same audit log. This is the property that justifies the schema-first approach: tools don't care which transport called them.

Figure 12.1 — MCP and in-app agents share one executor and one gate.

What MCP exposes beyond tools

lib/mcp/prompts.ts and lib/mcp/resources.ts surface two more MCP primitives:

  • Prompts — reusable parameterized prompts the host agent can invoke. Useful for "explain this code", "summarize this article" patterns where you want a prompt template, not a tool call.
  • Resources — addressable read-only data the host can subscribe to. The deck could expose thread://<id>/transcript or artifact://<id> as MCP resources.

The prompts/resources work is newer than the tools work and feels under-loved. The plumbing is there; the catalog of prompts and resources is thin.

Keep
The unified executor for in-app and MCP. This is the architectural prize. Don't fork it for "MCP-only sandboxing" or similar.
Extend
Populate the prompts and resources catalogs. Right now the deck exposes 35 tools and a thin prompts list. External agents would benefit from "give me the chat transcript", "list recent artifacts", "summarize this thread" prompts. Same investment pattern as the tools — schema-driven.
Chapter 13

The LLM router

The deck speaks four flavours of LLM provider behind one router: Anthropic, OpenAI, Google, and any OpenAI-compatible endpoint (Ollama, vLLM, llama-server, LM Studio, OpenRouter, Groq, DeepSeek). The slot system is the architectural beauty:

SlotEnv knobsTypical use
defaultLLM_PROVIDER, LLM_BASE_URL, LLM_MODELMain turn-taking
fastLLM_FAST_*Routing, classification, cheap calls
visionLLM_VISION_*Image input
embeddingLLM_EMBEDDING_*Vector store writes

Each slot is filled independently. A real configuration might use Groq for the fast slot (cheap classifies), Anthropic Sonnet for default (main turns), Qwen-VL on local Ollama for vision (privacy), and OpenAI text-embedding-3-small for embeddings. Four providers in one run, no special code paths.

Figure 13.1 — Four slots, four providers, one router. No special code paths.

What's missing

The router doesn't yet do automatic fallback. If the default slot fails, the agent gets an error — there's no "retry on the fast slot" or "fall back to local" logic. For a local-first cockpit, that's a missed opportunity: if your cloud key expires mid-conversation, the deck should be able to continue against a local model instead of failing.

Extend
Add per-slot fallback chains. LLM_MODEL=sonnet, LLM_FALLBACK=ollama://qwen2.5:7b. On rate-limit or auth error, transparent retry on the fallback. Surface the switch in the UI as a "router used X" tag on the message.
Watch
The "vision" slot. Vision needs aren't binary — sometimes a thumbnail's worth of image context is enough for the default model, sometimes you need a real vision model. Today every image-bearing turn goes through the vision slot. Consider a heuristic: small images → default, large/many → vision slot.
Chapter 14

Verdicts: what to extend, what to cut

Consolidated opinions from each chapter, ranked by leverage. Each row should be readable as a directive — "do this thing, this week, and the system gets better."

High-leverage extensions

#1 Extend
Unify voice into the chat path. Three-step plan exists. Step 1 alone removes the bug class behind every recent voice fix. Largest impact, lowest risk.
#2 Extend
Make the three allowlists derived from schema metadata. Annotate Zod schemas with .meta({ surfaces, risk, sideEffect }); generate bridgeToolList, manifest, mcpProfiles at build. Removes a permanent class of drift bugs.
#3 Extend
Add the missing universal tools. read_file, http_fetch, git, apply_patch. Each is half a day. Each eliminates clumsy execute_code workarounds.
#4 Extend
Two-axis risk classification. Reversibility ≠ dangerousness. execute_code in a sandbox is reversible; native_click on "Send" isn't. Treat them differently.
#5 Extend
Per-slot LLM fallback chains. Local-first means surviving cloud outages. LLM_FALLBACK env, transparent retry, surfaced in the UI.
#6 Extend
Schema-aware persistence projection. Each tool's "summary for replay" should be its own Zod schema. Replace the global "truncate at 200" rule.

Cuts

#1 Cut
The fullscreen voice overlay. Replaced by inline indicator in step 3 of the voice unification plan.
#2 Cut
Windows-only tools from non-Windows manifests. Eight schemas, all returning unsupported_platform. They confuse the LLM and inflate the catalog.
#3 Cut (deferred)
voice_* tables, if voice unification lands. Five tables for a feature about to fold into the chat path. After step 2 of the plan, most of these become dead weight.

Keep

Keep
Zod-first tool definitions. The keystone. Every architectural property of the system depends on it. Resist sirens that want to "make tools dynamic" or "load tools at runtime."
Keep
One executor, one gate, one event bus. The three rails. Anything new sits on top of these — never beside.
Keep
Events-as-source-of-truth in SQLite. Debugging without this would be brutal.
Keep
The workspace capability pattern. The cleanest way to expose pane operations to an agent. Don't shortcut.
Keep
Tool execution inside the Next.js process. Not in agent-ts. Co-location with persistence and the MCP server is the right boundary.

Watch

Watch
The 1305-line executor.ts. Fine at 40 tools. Split by category before it hits 80.
Watch
The native handle TTL. 60-second handle cache works for short loops, bites in long plans.
Watch
Two surface roots (/deck and /deck/workspace). Acceptable now as "simple vs power" but watch for divergence.
Watch
The optional ToolCallArgs wire event. Today's reducer back-fill works but is fragile. Either make it required or document the convention next to the type.
Watch
Voice-core process boundary. Reasonable when STT/TTS were Python-only; weaker case as cloud HTTP APIs improve.

E. Three rails to remember

The architectural keystone

One executor. One gate. One event bus.

Every tool, in-app or MCP, runs through lib/tools/executor.ts. Every side-effect goes through lib/approvals/gate.ts. Every event lands on the AG-UI hub and the events table.

Anything new sits on top of these three — never beside.
