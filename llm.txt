# Control Deck — LLM Context for Frontend Redesign

## Identity

Control Deck is a homelab AI control center — a unified dashboard for launching,
monitoring, and managing AI agents, models, generation pipelines, and services.
Not a chat app. Not a terminal UI. An operating layer for your entire machine and
AI stack. Think: Home Assistant for AI, Grafana that can execute, LM Studio for
everything, a flight deck.

## Tech Stack

- Next.js 16.0.10 (App Router) + React 19 + TypeScript
- Tailwind CSS 4.1.18 + Tailwind Animate
- shadcn/ui + Radix UI primitives
- Vercel AI SDK 5.0.113 (multi-provider abstraction)
- better-sqlite3 12.5.0 (local SQLite for threads, runs, artifacts, messages)
- Monaco Editor 0.55.1 (code editing in Canvas pane)
- @dnd-kit (drag-and-drop for dashboard widgets)
- Lucide React icons (16px, 1.5 stroke)
- Google Model Viewer (3D artifact rendering)
- Bun (package manager + dev server, port 3333)

## Provider SDKs

- @ai-sdk/openai, @ai-sdk/anthropic, @ai-sdk/google
- @ai-sdk/huggingface, @ai-sdk/openai-compatible
- Local: Ollama, llama.cpp server, vLLM, LM Studio (all via openai-compatible)
- Agent-GO server (external at localhost:4243) for orchestration

## External Services

- Agent-GO: http://localhost:4243 (agent orchestration + tool execution)
- ComfyUI: http://127.0.0.1:8188 (image generation workflows)
- Voice API: http://127.0.0.1:8000 (FastAPI sidecar for STT/TTS)
- VectorDB: http://localhost:4242 (Qdrant for RAG)
- SearxNG: http://localhost:8888 (web search)
- Ollama: http://127.0.0.1:11435

## State Management Architecture

### React Contexts

1. **DeckSettingsProvider** — preferences (model, theme, design system, voice settings),
   UI state (settingsOpen, railOpen, sidebarOpen), persisted to localStorage

2. **ThreadManagerProvider** — thread CRUD, active thread selection, message list,
   date-grouped thread list. Lifted to DeckShell level.

3. **CanvasProvider** — canvas open/close, editor content, execution output

4. **ChatInspectorProvider** — selected message for inspection, tool call + artifact data

### Key Hooks

- `useAgentRun` — **primary state machine** (reducer-based). Discriminated union RunPhase:
  `idle → submitted → thinking → streaming → executing → resuming → error`
  Drives timeline segment generation. Consumes SSE from /api/chat.

- `useThreadManager` — thread context consumer
- `useVoiceChat` — voice capture/playback (PTT, VAD, toggle modes)
- `useFileUploads` — file staging and persistence
- `useSystemStats` — polls /api/system/stats every 10s (GPU, services)
- `useModels` — fetches available model list (singleton cache)
- `useShortcuts` — global keyboard binding registry (priority-based)
- `useReducedMotion` — prefers-reduced-motion media query
- `useDashboardLayout` — widget grid order + visibility
- `useCanvas` — canvas panel toggle

### Run State Machine

```
IDLE ──→ SUBMITTED ──→ THINKING ──→ STREAMING ←──→ EXECUTING
                                      │                ↓
                                      │            RESUMING
                                      │                │
                                      ↓                ↓
                                    IDLE            (back to STREAMING)
                                      ↑
                                    ERROR (terminal)
```

Each phase drives: StatusStrip appearance, input disabled state, Stop button visibility,
timeline segment generation.

### Timeline Segments (discriminated union)

```typescript
type TimelineSegment =
  | { kind: "user-message"; ... }
  | { kind: "agent-reasoning"; ... }     // extended thinking, collapsible
  | { kind: "agent-activity"; steps: ActivityStep[]; ... }  // grouped tool calls
  | { kind: "agent-message"; ... }       // streamed text
  | { kind: "artifact"; ... }            // image/code/audio/3D
  | { kind: "error"; ... }               // error with retry
```

## Data Flow

### Chat Request Lifecycle

```
User submits message
  → POST /api/chat { messages, model, threadId }
  → Server selects provider (runtime override > env > defaults)
  → Server creates Run record in SQLite
  → POST {AGENTGO_URL}/runs (with LLM config + messages)
  → GET {AGENTGO_URL}/runs/{runId}/events (SSE stream)
  → Map Agent-GO events → AG-UI events
  → Save events to DB + publish to hub + write to response SSE
  → Client reads SSE → dispatches to useAgentRun reducer → updates timeline
  → On tool call: execute tool → encode result with smartEncode()
  → On completion: emit RunFinished → generate thread title async
```

### AG-UI Event Types

```typescript
// Lifecycle
RunStarted, RunFinished, RunError

// Text streaming
TextMessageStart, TextMessageContent (delta), TextMessageEnd

// Tool execution
ToolCallStart, ToolCallArgs, ToolCallResult

// Rich content
ArtifactCreated

// Metrics
CostIncurred (tokens + USD)

// Interrupts
InterruptRequested, InterruptResolved
```

### DeckPayload Envelope

```typescript
type DeckPayload =
  | { kind: "json"; data: unknown }
  | { kind: "glyph"; glyph: string }     // custom binary encoding
  | { kind: "text"; text: string }
  | { kind: "binary"; base64: string; mimeType: string }
```

## Component Patterns

### Message Rendering
- User messages: right-aligned with subtle indigo bg
- Agent text: flat left-aligned, no bubble
- Activity blocks: 2px left accent border, grouped tool steps, collapsible
- Reasoning: collapsible bubble, auto-collapsed when not streaming
- Artifacts: full-width card, max 400px height for images

### Status Strip
- Replaces blinking dot with proper phase communication
- Phases: thinking (accent pulse), streaming (accent), executing (warning),
  resuming (accent), error (red)
- Pulsing icon + label text

### Composer
- Context row: model badge + attachment previews
- Auto-growing textarea
- Action row: attach button, voice controls, send/stop button
- Send button morphs to Stop during active run

## For a Frontend Redesign

Any redesign should preserve:
1. The `useAgentRun` state machine and timeline segment architecture
2. The provider abstraction (slot-based multi-LLM)
3. The AG-UI event protocol and SSE streaming
4. SQLite persistence layer
5. Shell decomposition (DeckShell → Sidebar + ThreadSidebar + Body)
6. Context-based state (DeckSettings, ThreadManager, Canvas, Inspector)
7. All API routes (backend contract is stable)

- Visual tokens (colors, typography, spacing, timing, radii)

- Animation patterns
- Shell chrome appearance
- Widget design and dashboard layout
